# ADR-018: The voice channel is a full conversational loop — the realtime model speaks its replies and drives the UI through the same validated action pipeline

**Status:** Accepted · **Date:** 2026-06-02 · **Stretch:** no
**Supersedes:** the *Outputs* (TTS-out-only-for-explain-back) clause of [ADR-004](./ADR-004-modalities-and-sensors.md), the "voice is *not* used for content navigation" non-goal of [ADR-004](./ADR-004-modalities-and-sensors.md), and the *deferred full-duplex spoken reply* (Option D) of [ADR-016](./ADR-016-spoken-turns-and-tablet-touch.md) · **Refines:** [ADR-005](./ADR-005-adaptive-ui-runtime-contract.md), [ADR-006](./ADR-006-voice-and-agent-llm-stack.md) · **Related:** [ADR-010](./ADR-010-content-correctness-and-validation.md), [ADR-011](./ADR-011-evaluation-and-mastery-instrumentation.md), [ADR-015](./ADR-015-coherent-learning-surface-transcript.md) · **Superseded by:** none
**Contract:** yes — adds ONE append-only outbound `ServerMessage` kind (`transcript_stream`) carrying interim/final transcript chunks to the browser; constructs the production server-side realtime session + `VoiceBridge` (the previously-deferred live fill path); maps realtime tool calls onto the existing internal tactical menu. No reshaped existing payload; no new wire `Action` variant.

## Context

Voice in Polymath was designed across three ADRs and shipped as a set of seams that were never connected end-to-end:

- [ADR-004](./ADR-004-modalities-and-sensors.md) scoped voice to explain-back, anti-cheat, and conversational Q&A; chose a **narrow TTS-out channel used only for the explain-back prompt** ("all other system communication … is text"); and named "voice is *not* used for content navigation" a non-goal.
- [ADR-006](./ADR-006-voice-and-agent-llm-stack.md) chose OpenAI Realtime over LiveKit Agents for audio-native capture, with the realtime model on a *separate path* from the inner (text) agent.
- [ADR-016](./ADR-016-spoken-turns-and-tablet-touch.md) wired spoken Q&A as first-class **input** (server-captured, fed to the text agent, answered in **text**), and explicitly **deferred** full-duplex spoken reply (its Option D), citing three costs: it broadens the narrow TTS-out surface, it is a substantially larger build, and "a spoken-only reply leaves no transcript record unless separately transcribed."

What actually shipped: the browser `VoiceClient` joins a LiveKit room and publishes the learner's mic; the server **mints a token and nothing else**. No server-side participant joins the room, no `RealtimeSession` implementation exists (only `MockRealtimeSession`), the `LearnerUtteranceRegistry` stays empty, and `spoken_turn` correctly fails closed to an `ack`. The result a learner sees: the mic "connects," but no transcript appears, no reply appears, and the only visible state change is button text. The conversational tutor described across three ADRs does not exist in production.

Two things have changed in intent, which is why this is a superseding ADR rather than a build detail:

1. **The product target is a conversational tutor fused with live manipulatives** — a learner talks to the tutor, hears it talk back, and watches the workspace (truth tables, circuits, pseudocode) change as the conversation moves. A text-only reply with a silent room is not that product; it is a chat box with a dead microphone.
2. **The deferral conditions ADR-016 set have been met or are addressable now.** ADR-016 deferred spoken reply "once text-back is proven" and because spoken-only leaves no transcript. Text-back is shipped (F-30). And the central objection — no transcript record — dissolves under the realtime model's own design: it "emits both audio response capability **and an aligned text transcript stream we log per turn**" ([ADR-006](./ADR-006-voice-and-agent-llm-stack.md)). Streaming that same transcript to the UI gives us a durable, scrollable record of every spoken reply. The cost ADR-016 named is paid by a thing we were already logging.

## Decision

### 1. The realtime model is the conversational front of the tutor — it speaks its replies aloud

The TTS-out asymmetry of ADR-004 is superseded: the realtime tutor answers spoken turns in **audio** (full-duplex), not only text. This is gated by the same env/availability check as all voice (it is off, fail-closed, when LiveKit/OpenAI are unconfigured), so the text channel remains the universal floor — every spoken reply is **also** emitted as a transcript turn (below), so a learner who can't or won't use audio loses nothing.

The anti-cheat reasoning ADR-004 built on the *input* asymmetry is **unchanged**: the integrity-bearing modality is still the learner's *spoken input* (prosody, hesitation, restarts), captured server-side, and explain-back is **untouched** by this ADR — its registry, its five deterministic preconditions, and its LLM judge are exactly as shipped. Widening TTS *output* does not weaken the input signal; a learner still cannot paste an LLM answer into a short spoken explain-back window.

### 2. Every spoken turn — learner and tutor — streams to the UI as a transcript turn

A new append-only outbound `ServerMessage` kind, `transcript_stream`, carries `{ speaker: 'learner' | 'agent', text, final }` chunks over the existing app WebSocket. The browser renders interim (non-final) chunks as a single live, visually-distinct "in-progress" bubble, and on `final` commits the text as a durable `spokenTurn` in the append-only transcript ([ADR-015](./ADR-015-coherent-learning-surface-transcript.md)). The conversation the learner scrolls back through is the whole spoken dialogue, exactly as typed Q&A already is.

Transport choice: the app WebSocket, **not** the LiveKit data channel. The data channel is not wired on either side; the app WS already carries every `ServerMessage` through one Zod-validated seam. One transport, one capture discipline.

### 3. The realtime model drives the UI by *proposing* tactical moves — the guards remain the truth-maker

The realtime model is given **function-calling tools that mirror the internal tactical menu** (`@polymath/agent`'s `TacticalMove` vocabulary: `next_practice_item`, `simpler_item`, `rephrase`, `alt_representation`, `worked_example`, `answer_question`, `propose_hint`, `propose_transfer_probe`, `propose_mastery_transition`, `no_action`). A tool call is resolved server-side into a `TacticalMove`, then run through the **identical** pipeline the text agent's output traverses: `compileMove` → `validateOutboundAction` (Zod shape) → Layer-2 `claimedTruthTable` recompute ([ADR-010](./ADR-010-content-correctness-and-validation.md)) → `rejectUnauthorizedAction` (the earned-it semantic gate) → the XState guards.

This supersedes ADR-004's "voice is not for navigation" non-goal — but **preserves [ADR-005](./ADR-005-adaptive-ui-runtime-contract.md) verbatim.** ADR-005's constraint is on the *output shape* — "the LLM never emits component trees … it picks from a typed enum … the statechart is the grammar" — and on *who decides* — "the agent can *propose* mastery; the statechart guard is the truth-maker." The realtime model is **another proposer behind the same gate.** It cannot emit a new wire variant (the tools resolve only into the four locked `Action` shapes), it cannot mount UI we didn't design (it targets the typed registry kinds via the menu), and it cannot mint a privileged transition: a forged or premature `propose_transfer_probe`/`propose_mastery_transition` is downgraded to `no_action` exactly as the text agent's would be. The model proposes; the guards dispose.

### 4. Item scoring is unchanged; the model is *informed*, never the scorer

High-frequency correctness stays client-side and the BKT/streak/rule-gate fold stays server-derived from the full (bounded) event log ([ADR-008](./ADR-008-frontend-and-client-architecture.md), [ADR-011](./ADR-011-evaluation-and-mastery-instrumentation.md)) — this ADR does not move the correctness trust boundary. When a `submit` is folded during a live voice session, the server **pushes a compact lesson-state update into the realtime session as conversational context** (correct/incorrect, new BKT, streak, phase, hint level) so the model can react naturally ("nice — that's right; let's try a harder one") and choose its next tool call against the real state. The model never computes a score; it is told the score the server already computed.

## Options considered

**Reply modality — A: keep text-out, conversation feel comes from streaming text + live tiles** (ADR-016's shipped posture). Smaller build, no ADR conflict; rejected as the model of record because the product target is an audible tutor and the transcript-record objection that justified the original deferral no longer holds. **B: full-duplex spoken reply (chosen)** — the realtime model speaks; every reply is *also* a streamed transcript turn, so the durable record ADR-016 wanted is preserved.

**Who drives the UI from voice — C: voice answers only; all mounts/transitions stay on the submit-driven text path.** Smallest surface; rejected because a phase boundary mid-conversation ("you're ready — let's do a transfer check") would feel abrupt and disembodied if the voice tutor couldn't enact it. **D: the realtime model drives UI via tools through the existing guard pipeline (chosen)** — natural conversation, and the hardened earned-it gate already downgrades anything unearned, so the privileged proposals are safe.

**Two brains vs. one — E: route each finalized spoken utterance back through the text (LangGraph) agent to *also* decide UI.** Rejected: two LLMs answering the same turn duplicate and can contradict each other (the learner hears one answer, reads another), and the text agent would be inferring UI moves from conversational chatter it was not designed to consume. **F: the realtime model is the sole conversationalist and the sole proposer for voice turns (chosen)** — one brain talks and proposes; the guards (not a second brain) are the check.

**Transcript transport — G: LiveKit data channel.** Rejected: unwired on both client and server; a second transport with its own failure modes. **H: the existing app WebSocket via a new append-only `ServerMessage` kind (chosen)** — one Zod-validated seam, consistent with every other server→client message.

**Production realtime client — J: browser-direct OpenAI Realtime (ephemeral key, no server participant).** Rejected: it removes the server from the transcript path, breaking the "integrity input comes from a server-captured source, never the client frame" boundary that protects explain-back and spoken Q&A. **K: server-side LiveKit room participant bridging OpenAI Realtime (chosen)** — the server hears the room, so the captured transcript is server-owned; larger build, but it keeps the integrity boundary the rest of the system depends on.

## Consequences for the build

- **The deferred live fill path is now built.** A production `RealtimeSession` implementation (OpenAI Realtime API client) plus a **server-side LiveKit room participant** (receives the learner's mic track, publishes tutor audio) is constructed in `apps/agent/src/voice/`. `livekit-server-sdk` mints tokens but **cannot receive audio** — a server-side RTC client (`@livekit/rtc-node`) and the `openai` realtime client are new dependencies; both must build under Node 22 in the agent Docker image (native bindings — verify with an actual image build, per the Dockerfile-COPY/native-dep discipline in CLAUDE.md). The `RealtimeSession` **interface is the frozen seam** ([ADR-006](./ADR-006-voice-and-agent-llm-stack.md), F-10) — the production class *implements* it; it does not reshape it. `MockRealtimeSession` stays the test double.
- **Fail-closed everywhere voice already fails closed.** Unconfigured LiveKit/OpenAI → `503`/no bridge, exactly as today; the `spoken_turn` handler still acks-without-answering on an empty capture; the bridge fills `LearnerUtteranceRegistry.setLatest` **only on `t.final`** (never interim partials); the registry read stays consume-on-read; the socket binds its session from `session_start` only. This ADR adds capability, not new trust in the client.
- **The tool surface is the full menu, gated identically to the text path.** The realtime tool schema is kept in **lockstep** with `menu.ts`'s `TacticalMove` union and `openaiClient.ts`'s move enum (the `[...PREV_MENU, 'move']` discipline). A tool the model emits that resolves to a privileged action passes through `rejectUnauthorizedAction` unchanged.
- **One new contract kind, append-only.** `transcript_stream` is added to the `ServerMessage` union (source of truth `packages/contract/src/wire.ts`); its exhaustive consumer is the web message router (`apps/web/src/App.tsx`) and the WS client's `ServerMessageSchema` parse. No existing kind is reshaped; no `Action` variant is added.
- **Invariants preserved (unchanged):** ADR-005 (LLM never invents UI; guards are truth-maker); the five validation layers (ADR-010); the mastery gate's six conditions and fail-closed default (ADR-011); the distinct-variable cap on every server-side `equivalent()`/`truthTable()` over learner input; `events.app IS NULL` scoping on integrity reads; high-frequency interaction stays client-only; the statechart spine and the (now extended-by-one-kind) wire contract are otherwise untouched.
- **What this supersedes, precisely:** ADR-004's *Outputs* clause (TTS-out is no longer explain-back-only; spoken replies are allowed, always mirrored to a transcript turn) and ADR-004's "voice is not for navigation" non-goal (the voice model may *propose* navigation/UI moves, which the guards still gate). ADR-016's deferral of full-duplex spoken reply (Option D) is lifted. ADR-004's **input** asymmetry, anti-cheat thesis, the pulse, explain-back integrity, and ADR-016's touch contract all **stand**.

## Status note

Accepted as the architectural basis for the conversational voice-loop feature. The live audio round-trip (TTFT, barge-in latency, cache-hit cost) is verifiable only against real OpenAI Realtime over a real LiveKit room with credentials present — the offline/contract tests cover the seam, the tool→action→guard mapping, the transcript-stream contract, and the fail-closed behavior; the live smoke is run with keys set (extends `docs/voice-cross-platform-smoke.md`).

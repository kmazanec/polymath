# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Polymath** is a multimodal hyperresponsive mastery interface for Boolean logic. The pedagogical thesis: a learner has *mastered* a concept only when fluent across three irreducibly different representations — **truth tables, gate circuits, and pseudocode** — and that mastery is gated so strictly (BKT ≥0.95 + consecutive-correct + no-hints + response-time band + held-out transfer item + voice explain-back) that a "mastered" learner cannot have been pattern-matching or pasting LLM answers.

The architecture *is* the pedagogy. Read these before non-trivial work — they are the source of truth, not the code:
- `docs/ARCHITECTURE.md` — executive summary, data-flow diagrams, decision index.
- `docs/adrs/ADR-NNN-*.md` — every consequential decision **with its WHY**. ADRs are immutable after acceptance; a superseded decision gets a new ADR (`Supersedes: ADR-NNN`), never an edit.
- `docs/ROADMAP.md` — 7 iterations, 26 vertical-slice features, the iteration DAG, and the **contract change protocol**.
- `docs/features/NN-*.md` — per-feature specs. A feature ID like `F-05` maps to `docs/features/05-*.md`.

## Monorepo layout

pnpm workspace (`pnpm-workspace.yaml`), Node ≥22, ESM-only (`"type": "module"`). TypeScript project references off `tsconfig.base.json` (strict, `noUncheckedIndexedAccess`, `verbatimModuleSyntax`).

- `packages/contract` — `@polymath/contract`. Zod schemas + types for **all cross-cutting contracts**: `ComponentSpec` (the 12-variant component registry), `Action` (the agent's 4-variant wire output), the `ClientEvent`/`ServerMessage` WebSocket protocol, `PhaseName`, lesson-config schemas. **This is the most load-bearing package.** Imports are `.js`-suffixed (ESM).
- `packages/booleans` — `@polymath/booleans`. The single source of truth for Boolean correctness: `parse / evaluate / variables / truthTable / equivalent`. Pure, no deps. Its public signatures are **locked** (see the header comment in `src/index.ts`). Truth tables are MSB-first (first variable = most significant bit).
- `packages/statechart` — `@polymath/statechart`. The XState v5 lesson spine. Owns *when* the UI may change. The **phase shape** (`introducing → practicing → {hint, transferring} → assessed → {mastered, remediating}`) is locked; downstream features fill in guard *bodies* (`canDeclareMastery`, `canEndItem`), never re-shape the spine.
- `apps/agent` — `@polymath/agent`. Node HTTP + WebSocket service. LangGraph inner agent (stubbed until F-05), Drizzle/Postgres persistence, the server-side Action validation gate.
- `apps/web` — `@polymath/web`. Vite + React 19 + React Router SPA. Renders `ComponentSpec` via one **exhaustive switch** (`src/components/registry.tsx`) — no dynamic lookup, no `eval`, no `dangerouslySetInnerHTML`.
- `lessons/<id>/` — `content.json` + `mastery_config.json` per lesson, validated against the contract at load time.
- `infra/` (local + CI helpers: `deploy.sh`, `smoke.sh`, `caddy/`) and `ops/` (production compose + caddyfile synced to the droplet).

## Commands

Run from the repo root unless noted. Package manager is **pnpm** (via corepack).

```bash
pnpm install                       # install workspace deps
pnpm test                          # all unit/integration suites (vitest run, projects = packages/* + apps/*)
pnpm test:watch                    # watch mode
pnpm typecheck                     # tsc --noEmit across every package
pnpm build                         # build packages/** then apps/** (order matters: packages first)

# A single package's tests:
pnpm --filter @polymath/booleans test
pnpm --filter @polymath/agent test          # includes the WS+Postgres integration test (needs a DB — see below)

# A single test file / test name (vitest):
pnpm --filter @polymath/agent exec vitest run src/server.integration.test.ts
pnpm --filter @polymath/booleans exec vitest run -t "equivalent"

# Run the stack locally (Postgres + agent + web + Caddy, single entrypoint on :8080):
cp .env.example .env               # then fill secrets as features need them
docker compose up --build          # http://localhost:8080  (override port with CADDY_HOST_PORT)
./infra/smoke.sh                   # post-deploy smoke test (health, session, WS round-trip)

# Dev servers (without Docker):
pnpm --filter @polymath/web dev    # Vite dev server (:5173)
pnpm --filter @polymath/agent dev  # tsx watch; needs a reachable POSTGRES_URL

# Database (from apps/agent):
pnpm --filter @polymath/agent exec drizzle-kit generate   # new migration from schema.ts changes
pnpm --filter @polymath/agent run db:migrate              # apply migrations (also runs on agent boot)
```

The agent integration test needs Postgres. Locally it can spin its own; in CI it connects to a sibling container via `TEST_POSTGRES_URL` (the shell-executor test container has no Docker socket). If `@polymath/agent` tests hang or fail on connection, that's a missing/unreachable DB, not a code bug.

**A whole-workspace `pnpm test` shares ONE `polymath-test-pg` across the parallel project runners, and that shared DB is the #1 source of false integration failures — do not read it as a regression.** All agent suites set `fileParallelism:false` and serialize *within* the agent project, but the workspace runs the agent project *concurrently* with the web/contract/etc. projects, all pointed at the same container; under that contention DB-backed agent tests intermittently see `inserted 0 rows` / `expected N got 0` (and the `transfer_bank` seed in `runMigrations` swallows its error, so a contended seed silently leaves the bank empty and breaks downstream suites — `baseline transferItemCount expected 2 got 0`). The tell is **non-determinism**: a *different* set of DB-backed files fails on each full run. To confirm green-vs-flake, run the agent project ALONE (`pnpm --filter @polymath/agent test` — serial, owns the DB) and the non-agent projects separately; the union of those two isolated runs is the real result. This has bitten two iterations now (F-19 escalated it, I5 integration hit it); the durable fix (per-suite schema/DB isolation, or a retry/fail-loud transfer_bank seed) is a test-harness re-architecture, not a per-feature edit. Until then: isolated-run green is authoritative; a flaky full run is not a gating failure.

## How the pieces talk (the load-bearing invariants)

- **High-frequency interaction is client-only, and never touches the network.** Toggling truth-table inputs, dragging gates, the learner-triggered pulse animation, and the immediate correctness verdict (truth-table compare via `@polymath/booleans`, <5ms) all happen in the browser. The learner sees their answer marked correct *before* the agent decides what to mount next. This separation is what makes the UI feel alive — do not move correctness checking server-side.
- **The LLM is on the critical path only at phase boundaries** (~5–10 calls/lesson). It never invents UI. It picks a `kind` from the typed `ComponentSpec` registry and fills slots; the XState guards are the truth-maker. "Generative UI" in the freeform-JSX sense is an explicit non-goal (ADR-005).
- **The server never trusts the agent.** Every proposed `Action` is re-validated against the locked Zod schema before crossing the wire; a malformed action is downgraded to `no_action` (`apps/agent/src/agent/validateAction.ts`). Item-generating components carry a `claimedTruthTable` the server independently recomputes via `@polymath/booleans` (ADR-010 Layer 2).
- **The mastery gate fails closed.** A satisfied *sub*-condition is never mastery: a passed rule-gate or a passed transfer probe must NOT be treated as `mastered` while any required condition (`mastery_config.json` `requireExplainBackPass`, `requireHandCuratedTransfer`) is unmet or its mechanism is unbuilt. In I1, explain-back (F-11/F-12) doesn't exist, so the agent deliberately cannot reach `mastered` — a passed transfer goes to `no_action` (awaiting explain-back), and an exhausted/missing transfer bank is a degraded state, not a pass. When you add a gate condition, the default for a missing input is *block*, never *pass*.
- **Every server-side `equivalent()`/`truthTable()` on learner-controlled input needs the distinct-variable cap** (the booleans grammar permits 26 vars → 2^26 enumeration blocks the event loop). It's required on *all* such call sites — the transfer-verdict path AND the submit-correctness BKT path (`apps/agent/src/mastery/eventConsumer.ts`) — over-cap input is simply "incorrect", never an enumeration. A new call site that omits it is a DoS.
- **Outbound agent actions get a semantic gate, not just Zod + Layer-2.** Beyond shape (`validateAction.ts`) and `claimedTruthTable` recompute (Layer 2), the server rejects an action the learner hasn't *earned*: a `TransferProbe` mount is downgraded to `no_action` unless `ruleGatePassed` and the component matches an allowed unseen `transfer_bank` row (`apps/agent/src/server.ts`). This matters once an LLM provider is live and could fire an early/forged probe. Any future privileged action (mastery, lesson transition) needs the same earned-it check.
- **A privileged-action FAMILY must apply its earned-it gate to EVERY frame in the family, not just the obvious "enter" one — and the gate's predicate is the full precondition, not a partial.** F-26's playground gated `enter_playground` but left `playground_submit`, `playground_request_scaffold`, and `exit_playground` ungated, so a client that forged frames (or used the UI after a refused enter) got the whole ungraded capstone — *and* a forged `exit_playground` minted a `mount MasteryCelebration`, the production "declared mastered" signal the metrics fold, poisoning metric-6. When a feature adds a set of WS frames behind an earned-it check, gate **all** of them through one shared predicate (`playgroundEntryEarned`), and treat the **state-mutating frame that emits a trusted signal** (here, exit→celebration) as the highest-stakes one — never let it persist that signal unearned. The predicate must also be *complete*: the playground is the post-curriculum capstone, so "earned" is mastery **AND** the lesson is terminal (`loadLessonIfExists(lessonId+1) === undefined`) — gating on mastery alone let a forged frame open the capstone mid-curriculum. Mirror this on the client: mount the privileged UI only on the server's **accept** (its ack), never optimistically — an optimistic mount flashes a surface the wire refused. (MR !10 review; the build gated entry and the adversarial pass missed the three sibling frames.)
- **Server-derive integrity signals; don't trust the client flag or a capped history window.** Correctness (BKT/streak), hint level (`hintsByItem`), and the rule gate are all computed server-side from the full (bounded) event log — never from `submit.correct` (client) or the 5-event `recentHistory` window (which can mis-count and reset a ladder). A served hint counts; a refused one (during a probe) does not. **This rule binds EVERY persisted-fold consumer, not just the mastery gate — the counter-metrics dashboard is in scope too.** The F-21 metrics initially folded the client `payload.event.correct` flag for `dependency_check`/`visual_utility`, letting a scripted client send wrong answers with `correct:true` to manufacture a passing operator dashboard; the fix persists a server-recomputed `submitVerdict` at the submit turn (mirroring `transferVerdict`) and the metric reads that. Same family: a "declared mastered" signal must key on the **production** event (a persisted `mount MasteryCelebration` action, as `buildReport` does), NOT a transient `gateEvaluation.passed===true` that can be momentarily true on a turn that never celebrated (that inflated metric-6's false-positive denominator). When you add a metric/aggregate over `events`, fold the server-derived verdict, never the client frame. (MR !8 review.)
- **Lesson content must agree with the validator.** `loadLesson` (`apps/agent/src/lessons/loader.ts`) throws if a hand-authored `truthTable` disagrees with the computed one. The validator wins; fix the content.
- **The explain-back is the integrity boundary.** Deterministic preconditions (duration, word-count, KC-vocab, item-specific reference) run *first*; only if they pass does the LLM judge content (ADR-010/011).
- **An integrity input must come from a server-captured source, never the client frame — and a control that breaks the happy path is still a bug.** The explain-back transcript is read only from the server-side capture seam (`ExplainBackCaptureRegistry`, fed by the VoiceBridge), never from `event.transcript` on the WebSocket frame — a client holding a session UUID could otherwise POST a polished transcript and clear the voice gate without speaking (and pay for the judge). Absent a server transcript → empty → fails closed at precondition #3. The same fix had a second half worth internalizing: the *production* path must actually be wired to that server source (`createServer` injects `explainBackTranscriptFor`), or the security control makes mastery unreachable for a real learner. **When you fail an integrity input closed, verify the legitimate path still produces that input** — a gate nobody can pass is as broken as a gate anyone can forge. (MR !6 review; the adversarial panel caught neither half.)
- **A CLIENT-TRIGGERED server-captured signal must be consume-on-read (one-shot), and the seam's docs must not claim a fill path that doesn't exist.** F-30's spoken-turn path reads the server-captured utterance (never the client frame — same as explain-back), but the read was *non-consuming* and the trigger (`spoken_turn`) is **client-fired**: a client could replay `spoken_turn` indefinitely after one real utterance and re-answer the same stale text (re-billing the LLM, re-polluting the off-topic counter). The fix is `takeLatest` (read-AND-clear) so one capture answers exactly one turn. This refines the "server-captured source" rule: when the *learner* (not the server) decides when the captured value is consumed, the capture must be one-shot, or it is replayable — a forge-proof read is not enough if it's re-readable. Two siblings fell out of the same review: (a) feed the capture seam only from a **finalized** transcript segment (`t.final`), never interim ASR partials, or an early trigger answers an incomplete question; (b) **don't let a seam's comments overclaim** — the registry is *designed to be* filled by a server-side `VoiceBridge.onLearnerUtterance`, but that bridge is **not constructed in production** (it's the deferred `docs/voice-cross-platform-smoke.md` work, deferred identically for explain-back). A comment that says `createServer` "wires" a bridge it never constructs is a lie a reviewer will (rightly) flag as a gate-nobody-can-pass; say "designed to be filled by the deferred bridge; fails closed until then." (MR !11 review.)
- **A monotonic integrity counter must not be derived from the capped event-log window.** The off-topic / topic-guardrail count is computed from a *separate uncapped* count query (`countOffTopicAnswers`), not the bounded `MAX_SESSION_EVENTS` fold — otherwise a learner who trips the budget can push the offending rows out of the window with benign frames and drop back under it (a fail-open drift). This refines the "server-derive integrity signals" rule above: *bounded* is fine for tactical state, but any signal that must only ever accumulate needs an unbounded source and must fail closed if history is truncated before the aggregate is known. (MR !6 review.)
- **Dev/test seams are off-by-default behind an explicit opt-in env, not just `NODE_ENV`.** A seam that bypasses an integrity gate (e.g. `?testExplainBackVerdict`, `?testForce=mastered`) must require an explicit `POLYMATH_ENABLE_TEST_SEAMS=true` AND `NODE_ENV!=='production'` — gating on `NODE_ENV!=='production'` alone leaves it live in staging/preview/unset-NODE_ENV deployments. The integration harness sets the flag; production never does. A seam that injects a verdict must still run the deterministic preconditions, so it can't fold a pass on an empty payload. (Sibling of the env-gated-services fail-closed rule below; MR !6 review.)
- **Every Polymath integrity query over `events` must scope to `events.app IS NULL` (the D3 app discriminator).** The `app` column (`NULL`=polymath, `'baseline'`=F-16, and any future arm) was added so other apps' rows can share the `sessions`/`events` tables without polluting Polymath's metrics — but a query keyed on `sessionId` *only* would still fold a foreign-app row into the BKT/streak fold, the off-topic guardrail, or the recall throttle if a session id were ever shared across arms. Apply the filter at *every* integrity read site (`countOffTopicAnswers`, the bounded `deriveState` fold, `readRecalledKcs`), not just the ones where collision is reachable today — the discriminator is only protective if it's used uniformly. (Sibling of "server-derive integrity signals"; MR !7 review. Adding the column without using it in the reads is the trap.)
- **An application-level one-shot / uniqueness check is not atomic — back it with a DB constraint.** A "has this already been submitted?" SELECT-then-INSERT (e.g. the experiment pre/post/followup result writes) lets two concurrent requests both observe "no rows" and both insert, doubling the rows an aggregate then averages over. The app-level check is the fast path and the friendly error; the DB `UNIQUE` constraint is the *correctness* boundary — catch its `23505` and map it to the same 409. When a route enforces "exactly once," add the matching unique index in the same change. (MR !7 review.)

## Contract change protocol (read before touching `packages/contract`)

Contracts lock at the end of iteration 0 specifically so iterations 1–6 can fan out across parallel feature branches. Changing them ripples everywhere, so:

- The WebSocket event/message kinds and the `Action` wire union are **append-only** — never re-shape an existing kind's payload. New variants are added behind a version, per `docs/ROADMAP.md`. Adding a **new optional field** to an existing event is allowed (it doesn't break existing senders) — e.g. `submit.repSubmission`, the optional rep-native learner-input union added in I1 alongside the unchanged required `submission` string.
- Adding a `ComponentSpec` `kind` is a **coordinated change across three places**: the union in `packages/contract/src/component.ts`, the `COMPONENT_KINDS` array, the web renderer switch in `apps/web/src/components/registry.tsx` (the `never` default makes a missing case a compile error), and the agent's prompt+validator. Removals require a deprecation window.
- `@polymath/booleans` public signatures are locked; the gate alphabet may grow but the function shapes don't. **When the alphabet grows, check associativity in `astToExpression`, not just the parser/evaluator.** AND/OR are associative, so their AST→string rendering can flatten (`a AND b AND c`) and re-parse to an equivalent tree. NAND and NOR are **non-associative** (`A NAND (B NAND C) ≠ (A NAND B) NAND C`), and the parser is left-associative — so a flat render of a right-nested NAND/NOR tree re-parses to the WRONG grouping and silently mis-scores a learner's correct pseudocode/expression submission. A new gate must parenthesize its operands in `astToExpression` according to the gate's own associativity (the right operand of a left-associative non-associative op needs parens whenever it's at the same-or-lower precedence). Add round-trip tests (`parse(astToExpression(ast))` deep-equals `ast`) for right-nested and mixed-precedence cases, not just the flat `A OP B` case. (MR !9 review.)
- **A contract barrier that ships sibling-feature *implementation* code must ship it covered, or it reds a coverage-gated package for every feature built off the barrier — and that restoration becomes a convergence point.** `@polymath/booleans` enforces 100% coverage (`packages/booleans/vitest.config.ts`). When the I6 barrier (`build/i6-stretch`) landed F-26's `playgroundEquivalence.ts` and F-22's NAND/NOR grammar arms *frozen but untested*, the package dropped to 92% — so F-22, F-23, AND F-26 each independently *created* `playgroundEquivalence.test.ts` to get their own branch green. At integration these collided as an add/add conflict that had to be hand-merged (keeping the union of cases, incl. F-22's `vi.spyOn(equivalent)` catch-arm test that F-23's set lacked). Two rules fall out: (1) when a barrier freezes runnable code into a coverage-gated package, freeze its tests in the same commit so the gate is green at the barrier; (2) if multiple features must restore the same gate, expect a test-file convergence — resolve it by *unioning* the cases (a coverage gate is satisfied by the superset), never by taking one side. (I6 MR integration.)

## External-service config fails closed (env-gated integrations)

A feature gated on external-service env (LiveKit, OpenAI, PostHog, LangSmith, …) must treat a **partial** configuration as *not configured* and fail closed, never emit a half-valid success. The `POST /api/realtime/session` route is the pattern: it requires `LIVEKIT_API_KEY` **and** `LIVEKIT_API_SECRET` **and** a non-empty `LIVEKIT_URL` before minting — a set key/secret with an empty URL returns `503 "voice not configured"`, not a `201` carrying `url:""` that the browser can't use. When you wire the next env-gated service, validate *every* field the response depends on, not just one, and serve a clean 503 otherwise. (Sibling of the mastery "fails closed" invariant above: a missing input is *block*, never a degraded *pass*.)

**Operator/research routes on the public agent port need auth, and the secret fails closed.** The agent's HTTP routes all sit behind Caddy's public `/api/*` — anything that reads or mutates data keyed only by a UUID (the experiment subject routes, the session-replay route) is reachable by anyone who learns or guesses that UUID, unless it's gated. The pattern (`checkOperatorAuth`, MR !7): require `POLYMATH_OPERATOR_SECRET` via `Authorization: Bearer` / `X-Operator-Secret` (constant-time compare) on operator/debug routes; when the secret is **unset**, fail closed in production (`503`) and stay open only in dev/CI (`NODE_ENV!=='production'`), so the offline integration suite needs no secret. A learner-facing route that already carries its own per-request secret (the follow-up route's random token) is exempt. When you add a route that returns or mutates research/teaching data by id, gate it the same way — don't assume "the UUID is unguessable" is access control. (MR !7 review.)

**A WebSocket connection must BIND to its session via a validated `session_start` on THAT socket — never act on a sessionId parsed from an arbitrary frame.** The HTTP rule above has a WS sibling. F-19's server-side end-of-session detection scheduled data deletion on socket close for `boundSessionId`, but `boundSessionId` was set to whatever `sessionId` appeared on *any* inbound frame — so anyone who knew a victim's session UUID could open a socket, send one frame naming it, disconnect, and stamp the victim's events + `learner_state` for hard-deletion after the grace window (a **destructive DoS with no credentials**), and a stray frame on a reconnect/duplicate-tab mis-fired the deletion timer for an active session. The fix: bind the socket's session **only** from a `session_start` frame on that connection, bind **once**, never rebind to a different id mid-connection. Any destructive or privileged WS-triggered action (deletion scheduling, and by extension a future mastery/transition reflex) must key off that bound id, not a per-frame id. The same binding closes the lower-severity beacon-pollution path (`ui_mount`/`intelligibility_response` spammed into a victim's log). (MR !8 review.)

**Vite inlines `import.meta.env.VITE_*` at BUILD time, so any web env a feature depends on must be a `docker build` ARG — a runtime compose `env_file` never reaches the bundle.** F-20's PostHog config (`VITE_POSTHOG_KEY`/`VITE_POSTHOG_HOST`) was declared as `ARG`s in `apps/web/Dockerfile` but the prod `ops/compose.prod.yaml` `web.build` had no `args:` block, so the deployed bundle inlined blanks and analytics shipped permanently off even when `/etc/polymath/.env` set the keys. Wire the value through `web.build.args: { VITE_X: ${VITE_X:-} }` (the deploy passes `--env-file` to `compose ... up --build`, so compose interpolates it into the build). This is the build-vs-runtime sibling of the "verify against the actual `vite build` artifact" DCE rule below. (MR !8 review.)

**A UUID-exempt READ does not license a UUID-triggered MINT — separate reading learner-owned data from creating a durable credential.** The handoff route is the documented exemption: `GET /api/session/:id/handoff` returns the learner's own artifact keyed only by the session UUID (the `followup_token`-style learner-owned-shareable carve-out). But the bare GET *also auto-minted* a persistent public share token as a side effect of the read — so anyone who learned the UUID from a log/screenshot/history could create a permanent, publicly-fetchable share link without the learner's consent. The fix (MR !9): the read stays UUID-exempt and returns an *existing* link, but **minting moved to an explicit `POST .../handoff/share`** — creating a durable credential is an action, never a read side effect. Rule: when a route is exempt from auth because the data is "the learner's own," that exemption covers *reading* it; any step that *mints a lasting access credential* (a share token, an API key, a signed URL) is a separate privileged action and must be its own explicit, non-GET endpoint — a read must be safe to trigger by anyone holding the id. (MR !9 review; the F-24 plan's D24-3 signed off the read exemption but conflated it with the mint.)

## Deploy

Single DigitalOcean droplet (`ssh gauntlet`) behind a shared Caddy, live at **https://polymath.biograph.dev**. GitLab CI (`.gitlab-ci.yml`): `verify` (typecheck + non-agent tests + build) and `agent_test` (agent suite against a sibling Postgres) run on MRs; on a push to `main`, `deploy` runs `infra/deploy.sh` (release-symlink pattern, atomic swap, health-check with rollback, Drizzle migrations on agent boot). CI runs on a **shell executor** — jobs do real work inside `node:22` containers and must never write build artifacts back into the checkout dir (root-owned files there break the next pipeline's checkout). Postgres data lives at `/opt/polymath/postgres` on the host, outside the release tree.

- **If a feature reads a file at runtime, confirm it's COPYed into the Docker image.** `apps/agent/Dockerfile` copies a curated set of dirs (`apps/agent`, `lessons`, `seed_data`, …) — it is *not* the whole repo. A path that resolves in local dev / CI (which have the full checkout) can still be `ENOENT` in the deployed image, and since migrations + the transfer-bank seed run at boot, a missing file crashes the agent before it serves `/api/health`, so the deploy's health-check rolls back. Unit/diff review won't catch this (the file exists on disk); only an image build does. When adding a runtime-read data dir, add the matching `COPY` and verify with `docker build` + an `ls` in the image. Boot-time seeding should also be **non-fatal** (degrade to a read-only/empty bank, don't crash) so a bad data file is a degraded read path, not a total outage. **This applies to new workspace packages too:** the Dockerfile COPYs a curated `packages/*` set (manifest in the `deps` stage, source in `runtime`). Adding a `packages/<new>` the agent imports without adding both `COPY`s makes the in-image `pnpm install` fail `WORKSPACE_PKG_NOT_FOUND` and the agent crash at boot — a `docker build` is the only thing that catches it (F-09's `@polymath/bkt`).
- **Never inject a secret into an `merge_request_event` job that runs the MR's own code.** A same-project MR can modify a test, build script, or dependency hook to read an env var and exfiltrate it over the network (masking only hides accidental *log* output). MR pipelines stay **offline-only**: the `verify`/`agent_test` jobs do not receive `OPENAI_API_KEY` (or any provider secret). A key-gated step (e.g. the explain-back live ≥90% LLM-judge eval) runs in a **protected-branch or manual maintainer-only** job (`explain_back_live_eval`: auto on protected `main`, `when: never` on MR pipelines) where the protected variable is exposed only to trusted code. The offline half of such a check (preconditions-vs-labels) still gates MRs. (MR !6 review.)
- **Even in a trusted job, don't put a secret in a process's argv — pass it by name.** A protected `OPENAI_API_KEY` is safe from MR exfil (above), but `docker run -e OPENAI_API_KEY="$OPENAI_API_KEY"` puts the *value* on the container's command line, where any user/process with `ps` on the shared runner host can read it (GitLab masking only scrubs *logs*, not argv). Pass the **name only** — `docker run -e OPENAI_API_KEY …` — so Docker inherits the value from the job environment and it never appears in argv. Applies to every key-gated `docker run`/`exec` in `.gitlab-ci.yml` (the live-eval jobs), not just the one a reviewer happened to flag — fix the class. (MR !11 review.)
- **The inner agent's menu lives in two places that must stay in lockstep.** A tactical move (`apps/agent/src/agent/menu.ts` `TacticalMove` union + `compileMove`) is *also* an option the LLM provider must be able to emit — `apps/agent/src/agent/openaiClient.ts` has a structured-output schema enum + a `toTacticalMove` switch. Adding a move to the union without extending the provider makes the switch non-exhaustive (a typecheck error) and/or leaves the keyed LLM path unable to emit it. Add both; the `[...PREV_MENU, 'new_move']` pattern keeps the enum in lockstep.
- **Rep/workspace components must honor `spec.visibleReps` (the probe-integrity boundary).** A `ComponentSpec` carries `visibleReps`; a transfer probe (F-07) mounts a rep with the held-out reps excluded.
